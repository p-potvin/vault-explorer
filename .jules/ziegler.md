## 2024-05-28 - Shell Command Injection in Electron IPC Handlers
**Vulnerability:** The Electron IPC handlers (`get-everything-size` and `generate-webm`) used `child_process.exec` to execute `es.exe` and `ffmpeg`. File paths (`targetPath`, `itemPath`) passed from the frontend were directly interpolated into string commands, creating a severe shell command injection vulnerability.
**Learning:** `child_process.exec` spawns a shell to execute commands, meaning any unescaped shell metacharacters (like `;`, `&`, `|`) in user-provided input or filenames can be parsed as subsequent commands, executing arbitrary code with the privileges of the Electron backend.
**Prevention:** Always use `child_process.execFile` (or `child_process.spawn`) instead of `child_process.exec` when handling dynamic inputs or file paths. `execFile` executes the binary directly without spawning a shell, passing arguments as a safe array, which completely mitigates shell injection risks.
## 2026-05-08 - Shell Command Injection in Ad-Hoc Scripts
**Vulnerability:** Found another instance of `child_process.exec` in an ad-hoc script (`scripts/generate_webm.js`) passing unsanitized file paths.
**Learning:** Shell command injection risks are not isolated to Electron IPC handlers; ad-hoc helper scripts or batch processors running locally often handle untrusted paths and are equally vulnerable.
**Prevention:** Always use `child_process.execFile` in node scripts, including local utility scripts, whenever passing paths or dynamic input.
## 🛡️-05-14 - Secure IPC Handler Implementation & Error Handling
**Vulnerability:** The previously scaffolded `upscale-video` IPC handler lacked strict input validation and directly leaked system error messages (`err.message`) back to the renderer process.
**Learning:** Returning raw internal error strings to the frontend exposes internal application architecture and potential path structures. Furthermore, failing to validate input types (e.g., ensuring `itemPath` is a string) can crash the main process when passing it to Node `fs` or `path` functions.
**Prevention:** All IPC handlers accepting user input must strictly validate the expected type (e.g., `typeof arg === 'string'`) and existence of paths. Error handling must wrap the entire execution block in a `try...catch`, log the detailed stack trace internally using `console.error`, and return a generic, sanitized error object (e.g., `{ success: false, error: 'An error occurred during processing' }`) to the renderer.
## 🛡️-05-14 - Awaiting Promises in try...catch Blocks
**Vulnerability:** When returning a `new Promise` inside a `try...catch` block, failing to use the `await` keyword causes the function context to exit the `try...catch` block immediately. Synchronous exceptions (like `fs.mkdirSync` throwing an error) inside the promise executor result in an unhandled promise rejection that bypasses the catch block entirely, leaking raw error objects and file paths to the calling process/renderer.
**Learning:** A `try...catch` block does not automatically catch errors from asynchronously returned Promises unless the Promise is explicitly `await`ed before being returned.
**Prevention:** Always use `return await new Promise(...)` when relying on an outer `try...catch` block to sanitize errors. Furthermore, explicitly wrap synchronous filesystem calls inside the promise executor with their own try/catch to gracefully reject the promise instead of throwing.
